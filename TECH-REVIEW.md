## 1. 架构决策

### 1.1 推荐架构：单服务 + 内存状态

```
                    ┌─────────────────┐
                    │  Node.js Server  │
                    │  (Express +       │
                    │   Socket.io)      │
                    │                   │
                    │  - 游戏状态 (Map) │
                    │  - 房间逻辑       │
                    │  - 题目/验证      │
                    └─────────────────┘
                            │
              ┌─────────────┼─────────────┐
              │             │             │
           ┌──────┐     ┌──────┐      ┌──────┐
           │ 浏览器  │     │ 浏览器  │      │ 浏览器  │
           │(画师)  │     │(猜题者) │      │(猜题者) │
           └──────┘     └──────┘      └──────┘
```

**为什么不引入 Redis / 数据库？**

| 方案 | 复杂度 | 成本 | 适用规模 |
|------|--------|------|----------|
| 全内存 | ★☆☆ | $0 | < 100 房间 |
| Redis | ★★★ | $50+/月 | > 100 房间 |
| PostgreSQL | ★★☆ | $0-20/月 | 持久化需求 |

50人单房间的实际负载极低：
- 画师每秒最多60个 draw 事件
- 每个事件约 50-150 bytes
- 广播给 49 个客户端，服务器出口带宽约 **300KB/s**
- 内存占用：50玩家 + 画布数据 < 50MB

**结论**：Redis 和数据库在项目初期是过度设计。第一版用内存状态足矣。

### 1.2 服务端选型

| 选项 | 优势 | 劣势 | 推荐度 |
|------|------|------|--------|
| **Node.js** | 生态成熟、Socket.io 原生支持、调试友好 | 单线程 CPU 密集弱 | ★★★★★ |
| Bun | 启动快、高性能 | 生态年轻、Socket.io 兼容性存疑 | ★★☆ |
| Go | 高并发、WebSocket 性能好 | 开发效率较低、词库管理不如 JS | ★★★ |

**结论**：Node.js + TypeScript。Socket.io 4.x 对 Node 18+ 有完整支持，稳定性高。

### 1.3 前端选型

| 选项 | 优势 | 劣势 | 推荐度 |
|------|------|------|--------|
| **Vue 3 + Vite** | 响应式简洁、团队习惯优先 | 组件化生态稍弱于 React | ★★★★★ |
| 纯 HTML | 无构建依赖 | 无状态管理、难以维护 | ★★☆ |

**结论**：Vue 3 + Vite + TypeScript。Composition API + Pinia 状态管理，逻辑清晰。

### 1.4 Canvas 选型

| 选项 | 优势 | 劣势 | 推荐度 |
|------|------|------|--------|
| **Fabric.js** | 对象模型、鼠标事件、序列化、免费 | 包体积 200KB | ★★★★★ |
| Konva.js | React 绑定完善 | 商业授权（部分场景收费） | ★★★ |
| 纯 Canvas | 零依赖、性能最优 | 大量手写代码（触摸、历史、选区） | ★★☆ |

**结论**：Fabric.js。其对象模型直接映射绘图元素，省去大量底层代码。

### 1.5 实时通信

| 选项 | 优势 | 劣势 | 推荐度 |
|------|------|------|--------|
| **Socket.io** | 自动重连、房间隔离、心跳、fallback | 有协议开销（~1KB/消息头） | ★★★★★ |
| Raw WebSocket | 最小开销 | 需手写重连、房间、分片 | ★★★ |
| SSE | 简单、单向 | 不适合双向、需轮询 | ★★☆ |

Socket.io 的 ~1KB 协议头对于 50-150 byte 的绘图消息来说偏高，但在 60 msg/s 的量级下仅增加 **60KB/s**，完全可接受。其可靠性收益远超这点开销。

**结论**：Socket.io 4.x。

### 1.6 部署方案

| 平台 | 免费额度 | 持久化 | 推荐度 |
|------|---------|--------|--------|
| **Fly.io** | 3 个共享 VM | 支持 volume | ★★★★★ |
| Railway | 500h/月 | 支持 | ★★★★ |
| Render | 免费 100h | 支持 | ★★★ |
| Vercel | 无 WebSocket 支持 | — | ✗ |

Fly.io 的免费共享 VM 完全够用：单核 256MB，足够运行 Node.js + 10 个房间。

## 2. 性能分析

### 2.1 带宽计算（单房间 50 人）

假设场景：
- 画师以 60fps 绘图，每帧 1 个描边点（x, y, color, width）
- 每个点位约 50 bytes（JSON）

```
入口：60 × 50 = 3 KB/s（服务器接收）
广播：60 × 50 × 49 = 147 KB/s（服务器出口）
```

实际会更低，因为玩家不会持续不间断画直线，帧率远低于 60。

**结论**：50人房间的带宽需求在 150KB/s 以内，单实例轻松承载。

### 2.2 内存估算（单服务器多房间）

| 数据项 | 每玩家估算 | 50人房间 |
|--------|-----------|---------|
| 玩家信息 | ~500 bytes | 25 KB |
| 房间元数据 | — | 5 KB |
| 画布数据（Stroke[]） | ~2KB/s（绘图时） | 100 KB |
| **合计** | — | **< 200 KB/房间** |

10 个房间并发：约 2MB 内存，加上 Node.js 基础占用，总计 < 100MB。

**结论**：内存完全不是瓶颈。

### 2.3 潜在瓶颈与应对

| 瓶颈 | 触发条件 | 应对方案 |
|------|---------|---------|
| 绘图事件过密 | 画师疯狂快速绘制 | 客户端 16ms 限流 + 服务端节流 |
| 大量房间同时存在 | > 20 个房间 | 当前架构支持，暂无问题 |
| 多人同时聊天 | 所有人刷屏 | 每玩家 1 msg/s 限制 |
| 服务器崩溃 | — | 玩家看到错误提示后跳转首页（V1） |

## 3. 稳定性设计

### 3.1 断线重连策略

```
客户端                            服务端
  │                                │
  ├─── 连接 ────→                  │
  │                                │  分配 sessionId
  │←─── 成功 ────                  │
  │                                │
  │     [游戏进行中，断线]          │
  │                                │
  ├─── 重连 ────→                  │
  │    带 sessionId                 │
  │                                │  查询 sessionId
  │←─── 恢复房间状态 ────           │
  │                                │
```

实现要点：
- 连接时服务端分配 sessionId，存于房间玩家的 `sessionId` 字段
- 重连时携带 sessionId，服务端查找对应玩家，恢复状态
- 60 秒内未重连，视为离开，积分清空

### 3.2 服务端容错

| 场景 | 处理 |
|------|------|
| 服务器重启 | 内存数据丢失，房间关闭，玩家跳转首页（V2 里程碑：SQLite 持久化） |
| 内存泄漏 | 房间无人后主动清理，限制单玩家历史消息条数 |
| 异常崩溃 | `uncaughtException` 捕获 + 优雅退出 + Fly.io 自动重启 |

### 3.3 客户端健壮性

- Socket.io 自动重连（默认 1s/2s/4s... 指数退避）
- 画布加载失败：显示占位图 + 重试按钮
- 计时器不同步：服务器下发权威时间，客户端同步倒计时

## 5. 技术评审结论

### 5.1 最终技术栈

> 完整技术规范见 PRD.md §5.3「技术规范」，两文档以此为准。

| 层级 | 选择 |
|------|------|
| 后端运行时 | Node.js 18+ |
| 语言 | TypeScript |
| Web 框架 | Express.js |
| 实时通信 | Socket.io 4.x |
| 前端框架 | Vue 3 + Vite + Pinia |
| 画布库 | Fabric.js |
| 数据库 | 无（V1 全内存），SQLite（V2 里程碑） |
| 部署 | Docker + Fly.io |

### 5.2 架构权衡说明

**为什么不用 Bun？**
Bun 的启动速度和 HTTP 性能优于 Node.js，但 Socket.io 在 Bun 上的兼容性未经广泛验证。对于需要稳定运行的社交游戏，可靠性优先于启动速度。

**为什么不用 Redis？**
50人单房间 + 最多数十个房间的规模，内存 Map 完全够用。引入 Redis 徒增运维复杂度和成本。

**为什么用 Fabric.js 而不是原生 Canvas？**
绘图应用需要：对象拾取、变换、历史记录、序列化。纯手写这些功能需要 2000+ 行代码，Fabric.js 开箱即用，体积可控（gzipped ~80KB）。

**为什么不直接用 Canvas API？**
同上，手写成本高。

## 6. 风险与缓解

| 风险 | 等级 | 缓解措施 |
|------|------|---------|
| Socket.io 连接数上限 | 低 | 单实例支持万级连接，非瓶颈 |
| 50人同时绘图性能 | 低 | 差异化广播 + 客户端限流，架构上隔离 |
| 服务器内存泄漏 | 中 | 房间/玩家数据定期清理，限制消息队列长度 |
| 房间数据断电丢失 | 低 | V1 接受；V2 加 SQLite 持久化 |
| 敏感词过滤简陋 | 中 | V1 用简单黑名单；V2 接入Moderation API |

## 7. 调整建议（对 PRD 的微调）

基于技术评审，对 PRD 提出以下调整意见：

1. **断线重连时限**：原 PRD 写"60秒内重连" → **维持 60 秒**（合理）
2. **单实例上限**：原 PRD"单实例 ≥ 10 个房间" → **更新为 ≥ 50 个房间**（经计算更乐观）
3. **画布同步频率**：原 PRD 未定义 → **客户端限流 60fps，服务端 1 秒内最多处理 120 条事件**
4. **数据库时序**：原 PRD 未提及 → **V1 无持久化；SQLite 持久化作为 V2 里程碑**
5. **房间解散规则**：原 PRD 未定义 → **最后一人离开后 30 秒自动解散房间**
