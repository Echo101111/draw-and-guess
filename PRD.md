# 多人你画我猜 - 产品需求文档 (PRD)

> 本文档定义产品范围、边界情况与约束条件，作为开发决策的依据。

---

## 1. 产品概述

### 1.1 核心功能

一款支持 **50人同时在线** 的你画我猜游戏。玩家通过创建私密房间、分享邀请链接进行游戏。游戏过程中，画师在 Canvas 上绘画，其他玩家实时观看并提交答案，猜对者与画师同时得分。

### 1.2 目标用户

- 朋友聚会、团队建设等社交场景
- 线上教室、公司活动等多人互动场景

---

## 2. 房间系统

### 2.1 房间创建

| 字段 | 说明 | 约束 |
|------|------|------|
| 房间码 | 6位数字字母组合，唯一标识 | 系统生成，不可自定义 |
| 房间密码 | 加入房间的验证凭证 | 可选，1-20字符 |
| 房间名称 | 显示用，可重复 | 可选，1-20字符 |
| 最大人数 | 上限50人 | 默认50，房主可在创建时调整（5-50） |

**边界情况：**
- 房间码碰撞：生成时查重，重试3次仍冲突则报错（概率极低，< 10⁷⁶分之一）
- 房间已满：拒绝加入，返回 "房间人数已满" 提示
- 密码错误：返回 "密码错误" 提示，不暴露房间是否存在

### 2.2 玩家加入

- 方式一：输入房间码 + 密码（若有）直接加入
- 方式二：通过分享链接（URL 包含房间码）进入，显示密码输入框（如有密码）

**玩家命名：**
- 加入时输入昵称，1-10字符，不可为空
- 昵称唯一性：房间内不允许重复，后加入者需改名

**边界情况：**
- 昵称重复：提示 "该昵称已被使用，请换一个名字"
- 房间不存在：提示 "房间不存在或已结束"
- 游戏已开始：可正常加入作为观战者（见 3.4）

### 2.3 房主权限

- 创建房间者默认为房主
- 房主可踢出任意玩家（被踢者看到提示后跳转首页）
- 房主可设置房间密码、修改最大人数
- 房主可开始游戏（在 Lobby 中点击开始）
- 房主掉线：新加入的第一个玩家自动成为房主；若无人则房间保留但无人能操作

## 3. 游戏流程

### 3.1 回合结构

```
每个回合：
1. 服务器分配画师（轮流制，见 3.2）
2. 服务器从词库选取题目，推送给画师
3. 画师在 Canvas 作画，所有玩家实时看到
4. 其他玩家提交答案
5. 正确 / 超时 → 回合结束 → 下一回合
```

| 参数 | 默认值 | 可配置范围 |
|------|--------|-----------|
| 每回合时长 | 90 秒 | 60-180 秒 |
| 每局回合数 | 10 轮 | 5-20 轮 |

**边界情况：**
- 画师掉线：该回合直接结束，公布答案，算作超时处理
- 所有玩家已猜对：立即结束回合，剩余时间不延续到下回合

### 3.2 画师分配规则

- 按进入房间顺序轮流（先入先画）
- 已画过的玩家会跳过，回到队列末尾
- 每轮画师不能与上一轮相同（除非只有两人）
- 若只剩1名玩家可作画，强制结束游戏

### 3.3 题目系统

**词库来源：** 服务器内置词库，分类管理（动物、食物、日常用品等），每局游戏可配置启用哪些分类，默认全开。

**题目规则：**
- 每轮从词库随机抽取
- 每轮题目不同（同一局内不重复）
- 题目推送给画师，服务器存储明文，客户端仅接收显示文本

**难度梯度（可选）：** 词库按难度分级，房主可选择简单/普通/困难模式。

**边界情况：**
- 词库枯竭：重复使用已出过的题目，并从词库移除（直到重置）
- 画师放弃作画：计时归零视为超时

### 3.4 观战者

游戏开始后加入的玩家默认为观战者：
- 观战者可实时看到画布更新
- 观战者 **不得提交答案**
- 观战者可看到聊天和得分
- 观战者 **不参与积分**

观战者在游戏结束后（进入结算页面）可选择留在此房间，进入下一局（若房主开启）。

## 4. 答题与计分

### 4.1 答案验证

- 不区分大小写
- 去除前后空格
- 完全匹配方为正确

**边界情况：**
- 相似答案（如 "apple" vs "apples"）：视为错误
- 同一玩家短时间内重复提交相同答案：限流，3秒内不响应
- 玩家提交空答案或纯空格：忽略，不报错

### 4.2 得分规则

| 行为 | 得分 |
|------|------|
| 猜题者正确作答 | 基础 100 分 |
| 画师配合成功（至少1人答对） | 50 分 |
| 越早答对，额外奖励 | × (1 - 答题时间 / 总时间) |
| 画师本人作答 | 不允许（画师无输入框） |

**举例：** 第30秒答对，总时间90秒 → 基础100 + 奖励 100×(1-30/90) ≈ 133 分

### 4.3 积分展示

- 实时显示当前回合积分榜（猜对者高亮）
- 每回合结束展示该轮得分
- 游戏结束后展示完整排行榜

## 5. 绘图功能

### 5.1 工具集

| 工具 | 说明 |
|------|------|
| 画笔 | 默认工具，支持颜色和粗细 |
| 橡皮擦 | 擦除画布内容（等价于白笔） |
| 清空画布 | 一键清除（画师操作，需确认） |
| 颜色选择 | 预设12色 + 自定义色 |
| 粗细调节 | 3档（细/中/粗） |

**边界情况：**
- 多人同时操作画布：**禁止**，仅画师可操作，其他人 Canvas 为只读
- 画师切换时：旧画师画布锁定，新画师收到空白画布
- 清空画布确认：弹出确认对话框防止误触

### 5.2 画布同步

- 描边数据实时广播（增量传输，非快照）
- 传输内容：`[{x, y, color, width, type}]` 点位数组
- 观战者渲染有轻微延迟（< 200ms），可接受

**边界情况：**
- 网络波动导致丢包：使用 Socket.io 的自动重连 + 事件重传机制
- 画师绘图过快：客户端限流 60fps（每 16ms 最多发送1次），合并点位批量发送
- 服务端节流：每玩家每秒最多处理 120 条绘图事件，超出部分丢弃

### 5.3 技术规范

> 以下为 PRD 与技术评审文档共享的技术参数，修改任一方需同步更新。

| 参数 | 值 | 来源 |
|------|---|------|
| 前端框架 | Vue 3 + Vite + TypeScript + Pinia | 技术评审 §1.3 |
| 后端运行时 | Node.js 18+ + Express + Socket.io 4.x | 技术评审 §1.2/§1.5 |
| 画布库 | Fabric.js | 技术评审 §1.4 |
| 部署平台 | Docker + Fly.io（$0 免费 VM） | 技术评审 §1.6 |
| 数据库 | 无（V1 全内存），SQLite（V2 里程碑） | 技术评审 §5.1 |
| 画布客户端限流 | 60fps（16ms/帧） | 技术评审 §2.3 |
| 服务端事件上限 | 120 条/秒/玩家 | 技术评审 §2.3 |
| 断线重连时限 | 60 秒 | 技术评审 §3.1 |

## 6. 聊天与互动

### 6.1 聊天功能

- 全房间公开聊天
- 系统消息（玩家加入/离开/答对/回合开始等）自动推送，灰色显示
- 每条消息显示发送者昵称和时间戳

**边界情况：**
- 刷屏限制：玩家每秒最多发1条消息，超出提示 "发送太频繁"
- 敏感词过滤：简单的关键词黑名单（可扩展为 API）
- 空消息：忽略

### 6.2 答案框

- 猜题者专属输入框
- 回车提交
- 提交后清空输入框，允许继续提交
- 正确后输入框自动禁用（显示 "已猜对"），避免重复

## 7. 游戏状态

### 7.1 状态流转

```
Lobby (等待) ──房主开始──→ Playing (进行中)
                                    │
                              所有回合结束
                                    │
                              GameOver (结算)
                                    │
                              ────返回 Lobby
```

### 7.2 断线处理

| 场景 | 处理方式 |
|------|---------|
| 画师断线 | 回合强制结束，公布答案 |
| 普通玩家断线 | 断线后 60 秒内重连，保留昵称和积分；超时则视为离开 |
| 房主断线 | 新加入玩家接管；无人时房间冻结，显示 "房主已离开，等待中" |
| 服务器重启 | 内存数据丢失，房间关闭，玩家跳转首页（V2 里程碑：SQLite 持久化） |

### 7.3 房间生命周期

- 房间无超时机制，玩家离开不自动解散
- 最后一人离开后 **30 秒自动解散房间**（防止误退后房间残留）
- 游戏结束后房间保留，玩家可留在此 Lobby 进入下一局
- 房主可主动解散房间，所有人跳转首页

## 8. 非功能需求

### 8.1 性能目标

| 指标 | 目标 |
|------|------|
| 页面加载时间 | < 3 秒（3G 网络） |
| 画布同步延迟 | < 100ms（局域网） |
| WebSocket 消息吞吐 | 支持 50 人房间同时绘画 |
| 同时在线房间数 | 单实例 ≥ 50 个房间 |

### 8.2 兼容性

- 目标浏览器：Chrome 90+, Safari 14+, Firefox 88+
- 不支持 IE
- 支持桌面端（平板端暂不支持绘图功能）

### 8.3 安全约束

- 题目明文仅存服务端，客户端无法获取（除画师外）
- 答案验证在服务端完成
- 房间密码不在 URL 中明文传输
- 防刷：消息频率限制、API 限流

## 11. 代码规范与开发流程

| 参数 | 值 | 来源 |
|------|---|------|
| 前端框架 | Vue 3 + Vite + TypeScript + Pinia | 技术评审 §1.3 |
| 后端运行时 | Node.js 18+ + Express + Socket.io 4.x | 技术评审 §1.2/§1.5 |
| 画布库 | Fabric.js | 技术评审 §1.4 |

| 断线重连时限 | 60 秒 | 技术评审 §3.1 |

### 11.2 代码风格

| 规则 | 配置 |
|------|------|
| TypeScript 严格模式 | `strict: true`，禁止 `any` 隐式类型 |
| ESLint | `@antfu/eslint-config` 或 `eslint-config-standard-with-typescript` |
| Prettier | 单引号、printWidth 100、trailingComma: es5 |
| 组件文件命名 | PascalCase（`GameCanvas.vue`），工具函数 camelCase |
| 目录结构 | 按功能分层，不按文件类型分层 |

**禁止事项：**
- 禁止 `any`（无类型断言的裸使用）
- 禁止 `// @ts-ignore`
- 禁止魔法数字（常量抽取到 `src/shared/constants.ts`）
- 禁止直接修改 `props`（Vue 单向数据流）

### 11.3 Git 分支规范

```
main          ← 生产环境，稳定可部署
  └── dev     ← 开发主分支，所有功能合并至此
        ├── feat/room-system
        ├── feat/game-canvas
        └── fix/canvas-desync
```

- 分支命名：`feat/`、`fix/`、`chore/`
- 每完成一个小功能 → PR → review 后合并 dev
- 禁止直接 push 到 main

### 11.4 Review 流程

**规则：每个可独立运行的模块完成后必须 review，未 review 不合并。**

| 阶段 | 完成后动作 |
|------|-----------|
| Phase 0 项目初始化 | 创建 PR → 检查目录结构、类型定义（shared/）、ESLint 配置 |
| Phase 1 房间系统 | 创建 PR → 检查房间状态机、Socket 事件命名、边界处理 |
| Phase 2 游戏流程 | 创建 PR → 检查游戏逻辑、计分公式、Canvas 同步、词库数据 |
| Phase 3 完善稳定 | 创建 PR → 检查断线重连、观战者隔离、限流机制 |
| Phase 4 部署优化 | 创建 PR → 检查 Dockerfile、Secrets 硬编码、性能指标 |

详细步骤见 `DEVELOPMENT-PLAN.md`。

**Review 检查清单（每次必过）：**
- [ ] TypeScript 编译无错误（`tsc --noEmit`）
- [ ] ESLint 无 error（允许 warning）
- [ ] 核心逻辑有对应注释（WHY，非 WHAT）
- [ ] 边界情况已在 PRD 中覆盖，或已在 PRD 中新增说明
- [ ] 新增依赖已记录到文档

### 11.5 提交信息规范

```
<type>(<scope>): <subject>

feat(game): add round timer with countdown
fix(canvas): prevent double stroke on fast draw
chore(deps): upgrade socket.io to 4.7
docs(room): add room auto-dismiss rule (see PRD §7.3)
```
